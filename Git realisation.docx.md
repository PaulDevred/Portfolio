# Git – un pilier de ma rigueur de développement

**Ma vision de git**

Git est un outil de **versionnage** qui est aussi la base de toute collaboration efficace et de toute démarche de développement professionnelle. Il me permet de suivre précisément **l’évolution** d’un projet, d’expérimenter sans risque, et de revenir en arrière si nécessaire. Dans un contexte professionnel, il assure la **cohérence** du code entre plusieurs développeurs et garantit la **traçabilité** des modifications. Avec l’expérience, j’ai compris qu’un bon usage de Git repose autant sur la **technique** que sur la **rigueur collective** : messages de commit clairs, revues systématiques, et branches bien nommées. Aujourd’hui, je conseille de toujours définir un workflow clair dès le début d’un projet, même à petite échelle, pour éviter toute confusion ultérieure.

**Comment j’utilise Git dans mes projets personnels**  
Dès mes premiers projets, j’ai systématiquement utilisé Git pour structurer mon travail. Lors du développement de mes projets, notamment *Darwin Simulator*, je configure un dépôt Git, définis une convention de branches (main/dev/features), et met en place des pull requests pour assurer la relecture de code. Cela permet de collaborer en limitant les erreurs et garantit une meilleure qualité de livraison.

**Collaboration en entreprise (IoT Platform)**  
Chez IoThink-Solutions, Git est au cœur du processus de développement. J’ai participé à la création et à la maintenance de branches liées à différents projets (comme l’endPoint GRDF). Avec des revues de code, des merges, rebases (parfois interactifs), cherry-pick et revert réguliers, l’utilisation rigoureuse de DevOps a permis de permis de conserver nos repositories et leurs historiques cohérents. Chaque merge request fait l’objet d’une validation collective, garantissant la qualité et la traçabilité du code.

**Ce que je connais, et ce que je veux améliorer**

Je considère avoir un bon niveau d’autonomie sur Git : je maîtrise les commandes essentielles et avancées, ainsi que les workflows collaboratifs. Cependant, je vois encore une marge de progression sur les fonctionnalités plus poussées : intégration continue via DevOps et GitLab, gestion des sous-modules, hooks personnalisés par exemple.

**La suite de mon parcours avec Git**

À moyen terme, je souhaite renforcer mon expertise en intégrant Git dans des pipelines d’intégration et de déploiement continus (CI/CD). Mon objectif est de rendre les cycles de développement plus fiables et automatisés, en reliant directement Git aux processus de tests et de déploiement.  
Je prévois également de contribuer à des projets open source pour continuer à perfectionner mes pratiques collaboratives et me confronter à des workflows plus exigeants.
